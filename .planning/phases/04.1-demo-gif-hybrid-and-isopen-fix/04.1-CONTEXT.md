<decisions>
## Implementation Decisions

### A. isopen fallback strategy

- **D-01: Fallback przez `events(fig).window_open[]`** — Makie's documented public API dla "is the window still open" w nowych wersjach. Zwraca `Bool` z `Observable`. Bez konieczności sięgania po `GLMakie.to_native(...)` ani `GLFW.Window` bezpośrednio (mniejsza powierzchnia API, łatwiejszy fallback gdy używana CairoMakie w przyszłości). Zastosowane we wszystkich 3 call-sites (linie 202, 373, 377 `src/wizualizacja.jl`).

- **D-02: Helper `_is_window_open(fig)::Bool`** — wydzielić jedną private funkcję zamiast inline 3× ten sam pattern. Defensywna logika: jeśli `events(fig).window_open[]` rzuca (np. fig nie został wyświetlony przez `display()`), zwracamy `true` (continue loop) — alternatywa byłaby spowodować false-stop przed renderem pierwszej klatki. Komentarz polski wyjaśnia rationale.

- **D-03: Outer try/catch zawężony do prawdziwych init-time GLMakie błędów** — obecny `catch e` (linie 449-462) reklasyfikuje DOWOLNY `Exception` zawierający substring "GLMakie" w `sprint(showerror, e)` jako "display context issue". To maskuje runtime błędy (jak nasz MethodError). Fix: rzutować TYLKO `LoadError` z initialization-time loadu GLMakie LUB `ErrorException` zawierający w message "OpenGL"/"X11"/"display" (NIE "GLMakie" sam w sobie). MethodError MUSI propagować unwrapped.

### B. Hybrid GIF design (NN-construction → separator → SA)

- **D-04: Hybryda 3-fazowa** (decyzja użytkownika 2026-05-04, opcja C):
  - **Faza 1 (5s, ~150 klatek @30fps):** NN-construction edge-by-edge od pustej sceny. Punkty już widoczne, trasa rośnie po jednej krawędzi na klatkę (chunk size = `ceil(N/150)` ≈ 7 krawędzi/klatkę dla N=1000).
  - **Faza 2 (~0.5-1s, 15-30 klatek):** wizualny separator. Preferowane: czarna klatka + overlay text "Optymalizacja SA-2-opt" wyfade'owany. Alternatywne (decyzja planner'a): freeze last NN frame + tekst nakładający się 1s.
  - **Faza 3 (5s, ~150 klatek @30fps):** SA-2-opt optimization (obecne zachowanie skryptu, ale skrócone do połowy `liczba_krokow`). Zachować `T_zero=0.001` lock-in z plan 02-14.
- **D-05: Helper `animuj_nn_construction!(stan, fig, observable; chunk=7)`** — nowy plik LUB rozszerzenie `src/wizualizacja.jl`. Argumenty: `stan` (z pustą trasą lub `stan.trasa = [1]` start), `fig`, `obs_punkty_trasy` (Observable do mutacji), `chunk` (krawędzi per yield). Funkcja inkrementalnie buduje trasę NN (algorytm: greedy nearest unvisited z punktu `stan.trasa[end]`), po każdym `chunk` krawędzi mutuje Observable. Po zakończeniu `stan.trasa` to pełen cykl Hamiltona = identyczny output jak `inicjuj_nn!`. **Decyzja planner'a:** czy trzymać w `src/wizualizacja.jl` czy w nowym `src/animacja.jl`. Preferowane: nowy plik dla testowalności, ale `src/wizualizacja.jl` jest OK jeśli helper jest pure-internal (`_animuj_nn_...`).
- **D-06: Ponowne użycie `wizualizuj()` API** — `examples/eksport_mp4.jl` po refactorze wywoła `wizualizuj()` 2× (lub 1× nowy `wizualizuj_hybrid()` wrapper) w 1 `record()` block. Decyzja planner'a: czy wprowadzić nowy publiczny entry point czy zostawić skrypt-level orchestration. **Constraint:** nie modyfikować obecnego `wizualizuj()` signature ani semantyki — Phase 3 LOCKED.

### C. Resolution bump

- **D-07: `Figure(size=(1920, 960))` w `src/wizualizacja.jl:93`** (decyzja użytkownika opcja C3). Aspect ratio 2:1 zachowany (panel główny + overlay panel z `_setup_figure`). Test impact: Phase 3 testy nie sprawdzają rozmiaru — bezpieczne.
- **D-08: Parametryzacja przez kwarg `figure_size=(1920, 960)`** — preferowana zmiana semantyki: dodać kwarg `figure_size` do `wizualizuj()` z defaultem zachowującym backwards compat (`(1400, 700)`), `examples/eksport_mp4.jl` przekazuje `(1920, 960)`. ALTERNATYWNIE (decyzja planner'a, jeśli refactor zbyt inwazyjny): hardcoded bump `Figure(size=(1920, 960))` w `_setup_figure` — Phase 3 lock relaxed dla 4.1 z motywacją "regression fix" w SUMMARY. Preferowane: kwarg.
- **D-09: `SZEROKOSC_GIF=1600` w `examples/eksport_mp4.jl:26`** — Lanczos downscale z 1920 do 1600. Zostawia 16% nadmiaru sourcing'u dla anti-aliasing, plik docelowy ~600-1500 KB (akceptowalny dla README embed, GitHub <1.5 MB soft).

### D. Content-level smoke tests

- **D-10: Test "first frame ≤1 edge"** — po regen `assets/demo.gif`, dekodować pierwszą klatkę przez `FFMPEG_jll` lub `ImageMagick_jll` (jeśli dostępny w stdlib) i zliczyć non-background pixele tworzące linię. Decyzja planner'a: użyć `read(IOBuffer, ...)` na surowym GIF i parsować GIF89a header LUB spawn `ffmpeg -i demo.gif -vframes 1 first_frame.png` + `FileIO.load`. Preferowane: ffmpeg subprocess (mniej deps, FFMPEG_jll i tak transitive). Heurystyka "edges ≤1": liczba pikseli z dominantnym kolorem trasy < threshold (np. <500 px dla 1600×800 ramki).
- **D-11: Test "last frame energia < energia_nn"** — NIE robić frame-decoding, zamiast tego refactor `examples/eksport_mp4.jl` do funkcji zwracającej `(stan_final::StanSymulacji, energia_nn::Float64)`, wywołać w teście z mniejszym N=50 i `liczba_krokow=200` (smoke), assert `stan_final.energia < energia_nn`. Czysty test logiki, headless, szybki.
- **D-12: Live mode smoke z headless guard** — test `wizualizuj(..; liczba_krokow=10, kroki_na_klatke=2)` opakowany w `try/catch` na inicjalizacyjne błędy GLMakie (no display). Pattern: `Sys.iswindows() || haskey(ENV, "DISPLAY") || skip` LUB `try ... catch e; isa(e, LoadError) && occursin("OpenGL", sprint(showerror, e)) && return @test_skip; rethrow; end`. Preferowane drugie (działa na Windows nawet bez DISPLAY).
